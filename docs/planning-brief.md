# Football-Career-Ledger 规划简报

本文面向外部统筹规划 AI，用于评估插件下一阶段是否应加入半自动写入、能力成长建议和世界书规则桥接。

## 项目背景

`Football-Career-Ledger` 是面向 `funnycups/Luker` 的第三方前端插件。使用场景是足球模拟人生类长对话：用户通过模型推进剧情，但职业数据需要长期稳定保存。

当前痛点：

- 模型容易忘记或重复累计赛季进球、助攻、出场和合同状态。
- 自然语言上下文不适合保存精确职业履历。
- MVU / `stat_data` 适合保存当前场景变量，但不适合保存完整比赛和能力成长记录。
- 用户已有足球世界书 `足球.json`，其中包含 MVU 边界规则和球员能力评分锚点。

## 当前实现

插件已经实现：

- Luker Chat State 持久化，命名空间为 `football-career-ledger`。
- 手动录入球员、赛季、比赛、合同、财务、能力和杂项。
- JSON 导入导出、示例数据、清空数据和 API 自检。
- 只读公开 API，供其他插件读取账本。
- 提示词摘要注入，可把当前职业摘要作为最近上下文给写作模型参考。
- 赛季统计由结构化比赛数据实时汇总，不依赖模型计算。

重要代码位置：

- `src/schema.js`：顶层数据结构和迁移。
- `src/selectors.js`：确定性查询和赛季统计。
- `src/prompt.js`：提示词摘要和记忆图投影。
- `src/ui.js`：插件面板。
- `src/storage.js`：Luker Chat State 读写。
- `index.js`：插件初始化、设置、提示词注入和公开 API 注册。

## 数据分工建议

推荐分工：

- 插件账本：权威职业数据，保存结构化事实。
- 世界书：足球规则、时代背景、俱乐部资料、能力评分锚点。
- MVU / `stat_data`：当前时间、地点、球队、位置、存款等轻量现场变量。
- 写作模型：叙事、解释和提出待确认建议。
- 用户：确认关键数据是否写入。

不建议：

- 让模型直接维护赛季总数。
- 让 MVU 保存完整能力、比赛、合同、履历。
- 让模型直接写最终能力值。

## 赛季统计

当前已完成确定性赛季统计。插件从 `matches` 中筛选同一 `seasonId` 的比赛，然后实时汇总：

- `minutes > 0` -> 出场 +1
- `started === true` -> 首发 +1
- `goals` -> 个人进球累加
- `assists` -> 助攻累加
- `yellowCards` / `redCards` -> 牌数累加
- `rating` 为数字时参与平均评分

设计判断：赛季累计应继续留在插件中，不应交给记忆图或模型自然语言处理。

## 能力值问题

当前世界书 `足球.json` 中的“球员能力划分”条目提供：

- 综合能力层级。
- 速度、射门、传球、控球、防守、身体、意识七项解释。
- 单项评分区间。
- 参考球员和参考数值。
- 门将专属能力维度说明。

这些内容适合作为评分锚点，但不是可直接执行的公式。

当前插件能力维度：

- 速度
- 射门
- 传球
- 控球
- 防守
- 身体
- 意识

当前插件暂未实现：

- 门将专属能力 schema。
- 自动能力成长。
- 世界书规则自动读取。
- 模型建议队列。

## 推荐下一步方案

建议采用半自动、可确认流程：

1. 模型根据剧情和世界书规则提出结构化建议。
2. 插件把建议作为草稿展示。
3. 用户确认、修改或丢弃。
4. 插件写入结构化账本。

建议先做两类草稿：

- 比赛草稿：日期、赛季、赛事、球队、对手、比分、首发、分钟、进球、助攻、评分、重要比赛。
- 能力变化草稿：能力项、变化值、原因、来源比赛或训练事件。

这样可以利用模型理解叙事的能力，但避免模型直接成为权威数据源。

## 待决策问题

1. 是否需要自动从模型回复中抽取比赛草稿？
2. 草稿来源是读取最后一条 assistant 回复，还是要求模型输出专门的 `<football_ledger_suggestion>` 块？
3. 能力成长是否按事件触发，还是只在赛季阶段结算时建议？
4. 世界书能力评分锚点是否复制进插件文档/规则摘要，还是继续由模型通过世界书读取？
5. 是否应新增门将模式，包含扑救、反应、站位、高空球、出击、出球？
6. 是否需要赛季结束页，统一处理赛季总结、奖项、合同、转会和能力调整？
7. 是否需要为每个账本变更保存来源引用，例如来自哪轮对话、哪场比赛、哪个草稿？
8. 是否继续保持记忆图只读桥接，避免插件主动写入记忆图？

## 风险

- 自动抽取若没有确认环节，会把模型幻觉写入权威账本。
- 能力值若按单轮剧情直接增长，容易膨胀。
- 直接读取和解析世界书全文会增加耦合，并可能受用户世界书结构变化影响。
- 过早支持门将、伤病、疲劳、训练、转会等系统会扩大 schema 和 UI 复杂度。
- 如果插件主动写入 MVU 或记忆图，容易和现有世界书规则冲突。

## 保守路线

优先级建议：

1. 保持当前手动账本稳定。
2. 增加比赛草稿确认，不直接写入。
3. 增加能力变化草稿确认，不直接写入。
4. 将 `足球.json` 的能力评分条目整理为简短规则摘要，只用于提示词和 UI 说明。
5. 观察使用体验后，再决定是否做门将模式、赛季结算页和记忆图桥接。
